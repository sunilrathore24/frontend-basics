## 1. Overview & Philosophy

### What Is Module Federation?

Module Federation is a **webpack 5+ feature** (also supported by Vite via plugins) that allows
multiple independently built and deployed JavaScript applications to share code at **runtime**.
It is **not** an Angular feature, React feature, or Vue feature — it operates at the bundler level,
making it framework-agnostic by design.

### Core Terminology

| Term | Definition |
|------|-----------|
| **Host** | The shell application that loads and orchestrates remote applications. Owns the root injector, router, and application shell (navbar, sidebar, layout). |
| **Remote** | An independently built application that exposes one or more modules/components to be consumed by a host at runtime. |
| **remoteEntry.js** | The manifest file generated by webpack for each remote. Contains metadata about exposed modules and shared dependencies. The host fetches this at runtime. |
| **Shared Scope** | A webpack runtime mechanism that negotiates which version of a shared dependency (e.g., `@angular/core`) to use, avoiding duplicate bundles. |
| **Exposed Module** | A module explicitly declared in the remote's webpack config as available for consumption by the host. |
| **Container** | The webpack runtime wrapper around a remote application that manages its exposed modules and shared dependencies. |

### Why Module Federation?

```
Traditional Monolith:
┌─────────────────────────────────────────┐
│  Single Angular App                      │
│  - One repo, one build, one deployment   │
│  - All teams commit to same codebase     │
│  - One CI/CD pipeline                    │
│  - Merge conflicts, slow builds          │
│  - Full redeploy for any change          │
└─────────────────────────────────────────┘

Module Federation (Micro-Frontends):
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Team A   │  │ Team B   │  │ Team C   │
│ Own repo │  │ Own repo │  │ Own repo │
│ Own CI/CD│  │ Own CI/CD│  │ Own CI/CD│
│ Own deploy│ │ Own deploy│ │ Own deploy│
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
     └──────┬──────┘─────────────┘
            ▼
   ┌─────────────────┐
   │   Host (Shell)   │
   │   Stitches all   │
   │   at RUNTIME     │
   └─────────────────┘
```

## 2. Traditional Angular Bootstrap Flow

Understanding the traditional flow is essential before grasping how Module Federation modifies it.

```
main.ts
└── platformBrowserDynamic().bootstrapModule(AppModule)
      │
      ▼
Angular reads @NgModule metadata from AppModule
      │
      ├── declarations: [AppComponent, ...]
      ├── imports: [BrowserModule, RouterModule, ...]
      ├── providers: [AuthService, HttpInterceptors, ...]
      └── bootstrap: [AppComponent]
            │
            ▼
Creates the Root Injector (DI Container)
      │
      ├── Registers all providers from AppModule and imported modules
      ├── Sets up the Router
      └── Creates the platform (zone.js, change detection)
            │
            ▼
Bootstraps AppComponent
      │
      ▼
Renders the component tree into the DOM
      │
      └── <app-root>
            ├── <app-navbar />
            ├── <router-outlet>
            │     └── (lazy-loaded feature modules)
            └── <app-footer />
```

### Key Characteristics of Traditional Flow

- **Single platform instance**: One call to `platformBrowserDynamic()`
- **Single root injector**: All services registered at root level are singletons
- **Build-time bundling**: All code (including lazy modules) is bundled at build time
- **Single deployment artifact**: One `dist/` folder, one deployment

### Step-by-Step Bootstrap Sequence

#### Phase 1: Host Bootstrap (Identical to Traditional)

```
1. Browser loads index.html from host's server
2. main.ts executes
3. platformBrowserDynamic().bootstrapModule(AppModule)
4. Angular creates the root injector with all host-level providers
5. AppComponent renders the shell (navbar, sidebar, router-outlet, footer)
6. Router initializes and evaluates the current URL
```

#### Phase 2: Remote Module Loading (On Navigation)

```
7.  User navigates to /dashboard (or any route mapped to a remote)
8.  Angular Router encounters the route config with loadRemoteModule()
9.  Webpack runtime fetches remoteEntry.js from the remote's server/CDN
10. remoteEntry.js registers the remote container in webpack's shared scope
11. Shared dependency negotiation occurs:
    - Host declares: "I have @angular/core@16.2.0"
    - Remote declares: "I need @angular/core@^16.0.0"
    - Webpack resolves: "Use host's @angular/core@16.2.0" (singleton)
12. The exposed NgModule chunk(s) are downloaded
13. Angular creates a CHILD injector for the remote's feature module
    - Inherits from the host's root injector
    - Remote-specific providers are scoped to this child injector
14. Remote's components render inside the host's <router-outlet>
```

#### Phase 3: What Does NOT Happen

```
✗ Remote's main.ts is NOT executed
✗ Remote's platformBrowserDynamic() is NOT called
✗ No second Angular platform is created
✗ Remote's AppModule is NOT bootstrapped
✗ Remote's index.html is NOT loaded
✗ Shared dependencies are NOT downloaded again (reused from host)
```

### Host Router Configuration

```typescript
// host/src/app/app-routing.module.ts
import { loadRemoteModule } from '@angular-architects/module-federation';

const routes: Routes = [
  {
    path: '',
    component: ShellComponent,
    children: [
      // Local route (bundled with host)
      {
        path: 'home',
        loadChildren: () => import('./home/home.module').then(m => m.HomeModule)
      },

      // Remote route (fetched at runtime from Remote A's server)
      {
        path: 'dashboard',
        loadChildren: () =>
          loadRemoteModule({
            type: 'module',
            remoteEntry: 'https://dashboard.myapp.com/remoteEntry.js',
            exposedModule: './DashboardModule'
          }).then(m => m.DashboardModule)
      },

      // Remote route (fetched at runtime from Remote B's server)
      {
        path: 'reports',
        loadChildren: () =>
          loadRemoteModule({
            type: 'module',
            remoteEntry: 'https://reports.myapp.com/remoteEntry.js',
            exposedModule: './ReportsModule'
          }).then(m => m.ReportsModule)
      }
    ]
  }
];
```

### Host Webpack Configuration

```javascript
// host/webpack.config.js
const { ModuleFederationPlugin } = require('webpack').container;
const { shareAll } = require('@angular-architects/module-federation/webpack');

module.exports = {
  output: {
    uniqueName: 'host',
    publicPath: 'auto',
  },
  plugins: [
    new ModuleFederationPlugin({
      name: 'host',

      // Host does not expose anything (it's the consumer)
      remotes: {},

      // Share Angular and other common dependencies
      shared: shareAll({
        singleton: true,
        strictVersion: true,
        requiredVersion: 'auto'
      })
    })
  ]
};
```

### Remote Webpack Configuration

```javascript
// remote-dashboard/webpack.config.js
const { ModuleFederationPlugin } = require('webpack').container;
const { shareAll } = require('@angular-architects/module-federation/webpack');

module.exports = {
  output: {
    uniqueName: 'dashboard',
    publicPath: 'auto',
  },
  plugins: [
    new ModuleFederationPlugin({
      name: 'dashboard',

      // The manifest file that the host will fetch
      filename: 'remoteEntry.js',

      // Modules exposed to the host
      exposes: {
        './DashboardModule': './src/app/dashboard/dashboard.module.ts',
      },

      // Share the same dependencies with the host
      shared: shareAll({
        singleton: true,
        strictVersion: true,
        requiredVersion: 'auto'
      })
    })
  ]
};
```

## 4. Webpack Shared Scope & Dependency Negotiation

This is one of the most critical and least understood aspects of Module Federation.

### How Shared Scope Works

```
┌─────────────────────────────────────────────────────────────────┐
│                    WEBPACK SHARED SCOPE                          │
│                    (runtime, in browser memory)                  │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  @angular/core                                           │    │
│  │  ├── Host offers:  16.2.0 (loaded: true)                │    │
│  │  ├── Remote A needs: ^16.0.0 → ✅ uses host's 16.2.0   │    │
│  │  └── Remote B needs: ^16.0.0 → ✅ uses host's 16.2.0   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  rxjs                                                    │    │
│  │  ├── Host offers:  7.8.1 (loaded: true)                 │    │
│  │  ├── Remote A needs: ^7.0.0 → ✅ uses host's 7.8.1     │    │
│  │  └── Remote B needs: ^7.0.0 → ✅ uses host's 7.8.1     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  lodash (NOT shared as singleton)                        │    │
│  │  ├── Remote A bundles: 4.17.21 (own copy)               │    │
│  │  └── Remote B bundles: 4.17.15 (own copy)               │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

### Shared Configuration Options

```javascript
shared: {
  '@angular/core': {
    singleton: true,        // Only ONE instance across all containers
    strictVersion: true,    // Throw error if versions are incompatible
    requiredVersion: '^16.0.0',  // Semver range this container needs
    eager: false,           // Don't bundle into initial chunk (lazy load)
  },
  'lodash': {
    singleton: false,       // Each remote can have its own copy
    requiredVersion: false, // No version constraint
  }
}
```

### What Happens on Version Mismatch

```
Scenario: Host has @angular/core@16, Remote needs @angular/core@17

Case 1: strictVersion: true
  → RuntimeError: "Unsatisfied version 16.2.0 of shared singleton module
     @angular/core (required ^17.0.0)"
  → Remote FAILS to load
  → Host app continues working (graceful degradation possible)

Case 2: strictVersion: false
  → Warning in console
  → Remote uses host's v16 anyway
  → May work, may cause subtle runtime bugs
  → NOT RECOMMENDED for framework packages

Case 3: singleton: false
  → Remote downloads its own @angular/core@17
  → TWO Angular runtimes in the browser
  → TWO zone.js instances, TWO change detection systems
  → WILL cause bugs (services not shared, events not propagated)
  → NEVER do this for Angular
```

## 5. Dependency Injection Hierarchy in MFE

### Injector Tree in Module Federation

```
Platform Injector (Angular internals)
│
└── Root Injector (Host's AppModule)
    │
    ├── SharedAuthService          ← @Injectable({ providedIn: 'root' })
    ├── SharedHttpClient           ← provided by HttpClientModule
    ├── SharedRouter               ← provided by RouterModule.forRoot()
    ├── GlobalErrorHandler         ← provided in root
    │
    ├── Child Injector (Dashboard Remote - lazy loaded)
    │   ├── DashboardService       ← provided in DashboardModule
    │   ├── DashboardStore         ← provided in DashboardModule
    │   │
    │   │   (Can access SharedAuthService via injector hierarchy ✅)
    │   │   (Cannot access ReportsService ❌)
    │   │
    │   └── Grandchild Injector (Dashboard sub-module)
    │       └── WidgetService      ← provided in WidgetModule
    │
    └── Child Injector (Reports Remote - lazy loaded)
        ├── ReportsService         ← provided in ReportsModule
        ├── ReportsStore           ← provided in ReportsModule
        │
        │   (Can access SharedAuthService via injector hierarchy ✅)
        │   (Cannot access DashboardService ❌)
        │
        └── Grandchild Injector (Reports sub-module)
            └── ChartService       ← provided in ChartModule
```

### Singleton Service Pitfall

```typescript
// ⚠️ DANGER: If both host AND remote provide AuthService at root level

// Host's auth.service.ts
@Injectable({ providedIn: 'root' })
export class AuthService { /* host implementation */ }

// Remote's auth.service.ts (SAME class name, different file)
@Injectable({ providedIn: 'root' })
export class AuthService { /* remote implementation */ }

// Result: TWO AuthService instances exist
// Host components get Host's AuthService
// Remote components get Remote's AuthService
// User appears logged in on host but logged out on remote
```

### Solution: Shared Library Pattern

```typescript
// @myorg/shared-auth (published as npm package or federated shared module)
@Injectable({ providedIn: 'root' })
export class AuthService {
  // Single implementation used by all
}

// Both host and remote import from @myorg/shared-auth
// Webpack shared scope ensures only ONE instance exists
// webpack.config.js
shared: {
  '@myorg/shared-auth': { singleton: true, strictVersion: true }
}
```

## 6. Independent Deployment Model

### Deployment Architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│                     PRODUCTION INFRASTRUCTURE                        │
│                                                                      │
│  ┌──────────────┐                                                   │
│  │  Host Shell   │  https://app.mycompany.com                       │
│  │  CDN / Server │  Serves: index.html, main.js, polyfills.js      │
│  │               │  Does NOT contain any remote code                │
│  └──────┬───────┘                                                   │
│         │                                                            │
│         │  At runtime, browser fetches:                              │
│         │                                                            │
│  ┌──────▼──────────┐  ┌────────────────────┐  ┌──────────────────┐ │
│  │  Remote A        │  │  Remote B           │  │  Remote C        │ │
│  │  (Dashboard)     │  │  (Reports)          │  │  (Settings)      │ │
│  │                  │  │                     │  │                  │ │
│  │  CDN A / Server A│  │  CDN B / Server B   │  │  CDN C / Server C│ │
│  │                  │  │                     │  │                  │ │
│  │  Serves:         │  │  Serves:            │  │  Serves:         │ │
│  │  remoteEntry.js  │  │  remoteEntry.js     │  │  remoteEntry.js  │ │
│  │  chunk-abc.js    │  │  chunk-xyz.js       │  │  chunk-123.js    │ │
│  │  chunk-def.js    │  │  chunk-uvw.js       │  │  chunk-456.js    │ │
│  └──────────────────┘  └────────────────────┘  └──────────────────┘ │
│                                                                      │
│  Each remote is:                                                     │
│  ✓ Built independently (own webpack build)                          │
│  ✓ Deployed independently (own CI/CD pipeline)                      │
│  ✓ Versioned independently (own git repo & tags)                    │
│  ✓ Owned by a separate team                                         │
│  ✓ Hosted on separate infrastructure                                │
└─────────────────────────────────────────────────────────────────────┘
```

### Why Independent Deployment Works

The host references remotes by URL, not by bundled code. The connection is a **runtime HTTP fetch**, not a build-time import.

```
Build Time:                          Runtime:
┌──────────┐                         ┌──────────┐
│ Host     │  knows nothing about    │ Browser  │  fetches remoteEntry.js
│ webpack  │  remote's actual code   │          │  from remote's server
│ build    │                         │          │  when user navigates
└──────────┘                         └──────────┘

Host's dist/ folder contains ZERO bytes of remote code.
Remote's dist/ folder contains ZERO bytes of host code.
```

### Deployment Timeline Example

```
Week 1:
  Monday    - Host team deploys shell v2.1.0
  Wednesday - Dashboard team deploys dashboard v3.5.0
  Friday    - Reports team deploys reports v1.2.0

Week 2:
  Tuesday   - Dashboard team hotfixes dashboard v3.5.1
              (Host does NOT redeploy)
              (Reports does NOT redeploy)
              (Users get the fix on next navigation to /dashboard)

Week 3:
  Monday    - Reports team deploys reports v2.0.0 (major rewrite)
              (Host does NOT redeploy)
              (Dashboard does NOT redeploy)
              (Users see the new reports UI on next navigation to /reports)
```

### Standalone Mode for Development

Each remote can also run as a standalone application for local development:

```
remote-dashboard/
├── src/
│   ├── main.ts                    ← Standalone bootstrap
│   │   └── platformBrowserDynamic()
│   │       .bootstrapModule(AppModule)  ← Remote's own AppModule
│   │
│   ├── app/
│   │   ├── app.module.ts          ← Standalone shell for dev
│   │   ├── app.component.ts       ← Dev-only root component
│   │   └── dashboard/
│   │       ├── dashboard.module.ts  ← THIS is what gets exposed to host
│   │       └── dashboard.component.ts
│   │
│   └── bootstrap.ts               ← Dynamic import for shared scope init
│
├── webpack.config.js              ← ModuleFederationPlugin config
└── package.json
```

```typescript
// remote-dashboard/src/main.ts (standalone mode)
import('./bootstrap').catch(err => console.error(err));

// remote-dashboard/src/bootstrap.ts
import { platformBrowserDynamic } from '@angular/platform-browser-dynamic';
import { AppModule } from './app/app.module';

// This ONLY runs when the remote is loaded standalone (ng serve)
// When loaded inside the host, this file is NEVER executed
platformBrowserDynamic()
  .bootstrapModule(AppModule)
  .catch(err => console.error(err));
```

## 7. The Shared Contract Between Host & Remotes

### The Contract

For independent deployment to work, all teams must respect a shared contract:

```
┌─────────────────────────────────────────────────────────────────┐
│                    THE MFE CONTRACT                              │
│                                                                  │
│  1. EXPOSED MODULE NAME                                          │
│     The string in exposes: { './Module': '...' } must not change │
│     without coordinating with the host team.                     │
│                                                                  │
│  2. SHARED DEPENDENCY VERSIONS                                   │
│     All teams must use compatible versions of shared singletons  │
│     (Angular, RxJS, zone.js, etc.)                               │
│                                                                  │
│  3. REMOTE ENTRY URL                                             │
│     The URL where remoteEntry.js is served must remain stable    │
│     (or be dynamically resolved via config).                     │
│                                                                  │
│  4. ROUTE STRUCTURE                                              │
│     The remote's internal route structure should not break the   │
│     host's routing expectations.                                 │
│                                                                  │
│  5. SHARED SERVICE INTERFACES                                    │
│     If remotes consume host-provided services (AuthService,      │
│     etc.), the interface/API of those services must remain        │
│     backward compatible.                                         │
│                                                                  │
│  6. COMMUNICATION CONTRACTS                                      │
│     Event names, payload shapes, and shared state structures     │
│     must be versioned and documented.                            │
└─────────────────────────────────────────────────────────────────┘
```

### Contract Enforcement Strategies

```
Strategy 1: Shared npm package for interfaces
  @myorg/mfe-contracts
  ├── auth.interface.ts
  ├── event-bus.types.ts
  ├── shared-state.types.ts
  └── route-contracts.ts

Strategy 2: Schema validation in CI
  Before deploy → validate remoteEntry.js against host's expectations

Strategy 3: Contract testing (like Pact)
  Host publishes: "I expect Remote A to expose './DashboardModule'"
  Remote A CI verifies: "I do expose './DashboardModule'" → ✅ deploy

Strategy 4: Shared versions file
  shared-versions.json (in a shared repo)
  {
    "@angular/core": "~16.2.0",
    "@angular/router": "~16.2.0",
    "rxjs": "~7.8.0",
    "zone.js": "~0.13.0"
  }
  All teams' package.json must satisfy these ranges.
```

## 8. Multi-Framework Module Federation

Module Federation operates at the webpack level, not the framework level. This means remotes can
use any JavaScript framework — or no framework at all.

### Supported Tech Stacks for Remotes

```
Host (Angular) can load remotes built with:
├── Angular (any compatible version)
├── React / React 18+ with concurrent features
├── React + Next.js (with custom webpack config)
├── Vue 2 / Vue 3
├── Vue 3 + Nuxt (with custom webpack config)
├── Svelte / SvelteKit
├── Preact
├── Solid.js
├── Lit / Web Components
├── Vanilla JavaScript / TypeScript
├── Ember.js
├── Qwik
└── Any framework that webpack or Vite can bundle
```

### Multi-Framework Architecture Diagram


### The Universal Mount/Unmount Contract

When mixing frameworks, remotes cannot be loaded as Angular modules. Instead, each remote
exposes a framework-agnostic `mount` / `unmount` API:

```
┌──────────────────────────────────────────────────────────────┐
│  UNIVERSAL REMOTE CONTRACT (for cross-framework remotes)     │
│                                                               │
│  interface RemoteApp {                                        │
│    mount(                                                     │
│      element: HTMLElement,                                    │
│      options?: {                                              │
│        initialPath?: string;     // for sub-routing           │
│        sharedState?: any;        // for state passing         │
│        onNavigate?: (path: string) => void;  // route sync   │
│      }                                                        │
│    ): () => void;   // returns unmount/cleanup function       │
```

### Angular Host Wrapper Component (for React Remote)

```typescript
// host/src/app/wrappers/react-wrapper.component.ts
import {
  Component, ElementRef, ViewChild,
  AfterViewInit, OnDestroy, Input
} from '@angular/core';
import { loadRemoteModule } from '@angular-architects/module-federation';
import { Router, ActivatedRoute } from '@angular/router';

@Component({
  selector: 'app-react-wrapper',
  template: '<div #remoteContainer class="remote-container"></div>',
  styles: [':host { display: block; width: 100%; height: 100%; }']
})
export class ReactWrapperComponent implements AfterViewInit, OnDestroy {
  @ViewChild('remoteContainer', { static: true }) container!: ElementRef;
  @Input() remoteEntry!: string;
  @Input() exposedModule!: string;

  private unmountFn?: () => void;

  constructor(
    private router: Router,
    private route: ActivatedRoute
  ) {}

  async ngAfterViewInit() {
    try {
      const remoteModule = await loadRemoteModule({
        type: 'module',
        remoteEntry: this.remoteEntry,
        exposedModule: this.exposedModule
      });

      // Remote exposes a mount function
      this.unmountFn = remoteModule.mount(this.container.nativeElement, {
        initialPath: this.route.snapshot.url.join('/'),
        onNavigate: (path: string) => {
          // Sync remote's internal navigation back to host router
          this.router.navigateByUrl(path);
        }
      });
    } catch (error) {
      console.error('Failed to load remote:', error);
      // Render fallback UI
      this.container.nativeElement.innerHTML =
        '<p>Failed to load module. Please try again.</p>';
    }
  }

  ngOnDestroy() {
    // Critical: clean up the remote's framework instance
    this.unmountFn?.();
  }
}
```

## 9. Cross-Framework Communication Patterns

Since framework-specific DI and context don't cross boundaries, teams need framework-agnostic
communication mechanisms.

### Pattern 1: Custom Event Bus (Recommended)

```typescript
// @myorg/shared-event-bus (npm package consumed by all remotes)

type EventCallback<T = any> = (data: T) => void;

class EventBus {
  private listeners = new Map<string, Set<EventCallback>>();

  emit<T>(event: string, data: T): void {
    this.listeners.get(event)?.forEach(fn => fn(data));
  }

  on<T>(event: string, callback: EventCallback<T>): () => void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(callback);

    // Return unsubscribe function
    return () => {
      this.listeners.get(event)?.delete(callback);
    };
  }

  once<T>(event: string, callback: EventCallback<T>): () => void {
    const wrapper: EventCallback<T> = (data) => {
      callback(data);
      unsubscribe();
    };
    const unsubscribe = this.on(event, wrapper);
    return unsubscribe;
  }

  clear(event?: string): void {
    if (event) {
      this.listeners.delete(event);
    } else {
      this.listeners.clear();
    }
  }
}

// Singleton — shared across all remotes via webpack shared scope
export const bus = new EventBus();
```

Usage across frameworks:

```typescript
// Angular remote (publisher)
import { bus } from '@myorg/shared-event-bus';

@Component({ /* ... */ })

  selectUser(user: User) {

function UserDetail() {

}

  }
```

### Pattern 3: Shared State Store (Vanilla JS)

```typescript
// @myorg/shared-store
type Listener<T> = (state: T) => void;

export function createStore<T>(initialState: T) {
  let state = initialState;
  const listeners = new Set<Listener<T>>();

  return {
    getState: () => state,
    setState: (partial: Partial<T>) => {
      state = { ...state, ...partial };
      listeners.forEach(fn => fn(state));
    },
    subscribe: (listener: Listener<T>): (() => void) => {
      listeners.add(listener);
      return () => listeners.delete(listener);
    }
  };
}

// Usage

  user: null as User | null,
  theme: 'light' as 'light' | 'dark',
  locale: 'en',
});
```

## 10. Dynamic Remote Resolution

Hardcoding remote URLs in the host's router config creates a deployment coupling. Dynamic
resolution eliminates this.

### Architecture

```
┌──────────┐     ┌──────────────────┐     ┌──────────────────┐
│  Host    │────►│  Config Service   │────►│  Remote Registry │
│  App     │     │  (API / CDN)      │     │  (JSON / DB)     │
└──────────┘     └──────────────────┘     └──────────────────┘
     │                                           │
     │  GET /api/mfe-config                      │
     │  Response:                                │
     │  {                                        │
     │    "dashboard": {                         │
     │      "remoteEntry": "https://cdn-a.com/remoteEntry.js",
     │      "exposedModule": "./DashboardModule", │
     │      "routePath": "dashboard"             │
     │    },                                     │
     │    "reports": {                           │
     │      "remoteEntry": "https://cdn-b.com/remoteEntry.js",
     │      "exposedModule": "./ReportsModule",  │
     │      "routePath": "reports"               │
     │    }                                      │
     │  }                                        │
     ▼                                           │
  Router config built dynamically at runtime     │
  No host redeploy needed when remotes change    │
```

### Implementation

```typescript
// host/src/app/services/mfe-config.service.ts
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { firstValueFrom } from 'rxjs';

export interface RemoteConfig {
  remoteEntry: string;
  exposedModule: string;
  routePath: string;
  displayName: string;
  framework?: 'angular' | 'react' | 'vue' | 'vanilla';
}

@Injectable({ providedIn: 'root' })
export class MfeConfigService {
  private config: Record<string, RemoteConfig> = {};

  constructor(private http: HttpClient) {}

  async loadConfig(): Promise<void> {
    this.config = await firstValueFrom(
      this.http.get<Record<string, RemoteConfig>>('/api/mfe-config')
    );
  }

  getRemotes(): RemoteConfig[] {
    return Object.values(this.config);
  }

  getRemote(name: string): RemoteConfig | undefined {
    return this.config[name];
  }
}
```

```typescript
// host/src/app/app-routing.module.ts
import { APP_INITIALIZER } from '@angular/core';
import { Router, Routes } from '@angular/router';
import { loadRemoteModule } from '@angular-architects/module-federation';
import { MfeConfigService, RemoteConfig } from './services/mfe-config.service';

// Build routes dynamically after config is loaded
function buildDynamicRoutes(
  router: Router,
  configService: MfeConfigService
): () => Promise<void> {
  return async () => {
    await configService.loadConfig();

    const remoteRoutes: Routes = configService.getRemotes().map(remote => ({
      path: remote.routePath,
      loadChildren: () =>
        loadRemoteModule({
          type: 'module',
          remoteEntry: remote.remoteEntry,
          exposedModule: remote.exposedModule,
        }).then(m => m[Object.keys(m)[0]]) // Get the first exported module
    }));

    // Reset router config with dynamic routes
    router.resetConfig([
      ...router.config.filter(r => r.data?.['static']), // Keep static routes
      ...remoteRoutes,
      { path: '**', redirectTo: 'home' }
    ]);
  };
}

@NgModule({
  providers: [
    {
      provide: APP_INITIALIZER,
      useFactory: (router: Router, config: MfeConfigService) =>
        buildDynamicRoutes(router, config),
      deps: [Router, MfeConfigService],
      multi: true,
    }
  ]
})
export class AppRoutingModule {}
```

```

  If green fails health check → traffic stays on blue
  If green passes → switch traffic to green
```

## 13. Error Handling & Resilience Patterns

### What Can Go Wrong

```
1. Remote's server is down → remoteEntry.js fetch fails (HTTP 5xx)
2. Remote's build is broken → remoteEntry.js exists but chunks fail to load
3. Version mismatch → shared scope negotiation fails
4. Remote's exposed module name changed → import fails
5. Network timeout → slow CDN or user on poor connection
6. CORS misconfiguration → browser blocks remoteEntry.js fetch
7. Remote throws runtime error → unhandled exception in remote's component tree
```

### Resilience Architecture

```typescript
// host/src/app/services/remote-loader.service.ts
import { Injectable, ErrorHandler } from '@angular/core';
import { loadRemoteModule } from '@angular-architects/module-federation';

interface LoadOptions {
  remoteEntry: string;
  exposedModule: string;
  fallbackComponent?: any;
  retryCount?: number;
  retryDelay?: number;
  timeout?: number;
}

@Injectable({ providedIn: 'root' })
export class RemoteLoaderService {

  async load(options: LoadOptions): Promise<any> {
    const {
      remoteEntry,
      exposedModule,
      retryCount = 3,
      retryDelay = 1000,
      timeout = 10000
    } = options;

    for (let attempt = 1; attempt <= retryCount; attempt++) {
      try {
        const module = await Promise.race([
          loadRemoteModule({
            type: 'module',
            remoteEntry,
            exposedModule,
          }),
          this.createTimeout(timeout)
        ]);
        return module;
      } catch (error) {
        console.warn(
          `Failed to load remote (attempt ${attempt}/${retryCount}):`,
          error
        );

        if (attempt < retryCount) {
          await this.delay(retryDelay * attempt); // Exponential backoff
        } else {
          console.error('All retry attempts exhausted for remote:', remoteEntry);

          if (options.fallbackComponent) {
            return { default: options.fallbackComponent };
          }
          throw error;
        }
      }
    }
  }

  private createTimeout(ms: number): Promise<never> {
    return new Promise((_, reject) =>
      setTimeout(() => reject(new Error(`Remote load timeout after ${ms}ms`)), ms)
    );
  }

  private delay(ms: number): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}
```

```typescript
export class MfeErrorHandler implements ErrorHandler {
  handleError(error: any): void {
    // Detect if error originated from a remote
    if (this.isRemoteError(error)) {
      console.error('[MFE Error]', error);
      // Report to monitoring (Sentry, Datadog, etc.)
      // Optionally navigate to fallback
    } else {
      // Default Angular error handling
      console.error(error);
    }
  }

  private isRemoteError(error: any): boolean {
    const message = error?.message || '';
    return (
      message.includes('Loading chunk') ||
      message.includes('remoteEntry') ||
      message.includes('shared module')
    );
  }
}
```

### Performance Optimization Strategies

```
1. PRELOADING
   - Preload remoteEntry.js for likely-next-navigated routes
   - Use <link rel="prefetch"> in index.html for critical remotes
   - Preload on hover over navigation links

2. CACHING
   - remoteEntry.js: short cache (5 min) — must pick up new deployments
   - Feature chunks: long cache with content hash in filename
   - Shared chunks: long cache (Angular runtime doesn't change often)

3. SHARED DEPENDENCY OPTIMIZATION
   - Mark large shared deps as eager: false (lazy loaded)
   - Use requiredVersion: 'auto' to minimize version conflicts
   - Audit shared scope to avoid sharing unnecessary packages

4. LAZY LOADING WITHIN REMOTES
   - Remotes should also internally lazy-load their sub-features
   - Don't expose a single massive module — split into sub-routes

5. CDN STRATEGY
   - Host each remote on a CDN close to users
   - Use HTTP/2 or HTTP/3 for parallel chunk loading
   - Enable Brotli compression for all JS assets
```

## 15. Security Considerations

### Threat Model

```
┌─────────────────────────────────────────────────────────────────┐
│  THREAT                          │  RISK    │  MITIGATION       │
├──────────────────────────────────┼──────────┼───────────────────┤
│ Compromised remote serves        │ CRITICAL │ Subresource       │
│ malicious remoteEntry.js         │          │ Integrity (SRI),  │
│                                  │          │ CSP headers        │
├──────────────────────────────────┼──────────┼───────────────────┤
│ Man-in-the-middle replaces       │ HIGH     │ HTTPS everywhere, │
│ remote chunks in transit         │          │ HSTS headers      │
├──────────────────────────────────┼──────────┼───────────────────┤
│ Remote accesses host's DOM       │ MEDIUM   │ Shadow DOM        │
│ or global state maliciously      │          │ isolation,        │
│                                  │          │ sandboxed iframes │
├──────────────────────────────────┼──────────┼───────────────────┤
│ Remote reads auth tokens from    │ HIGH     │ HttpOnly cookies, │
│ localStorage                     │          │ token scoping     │
├──────────────────────────────────┼──────────┼───────────────────┤
│ Dependency confusion attack      │ MEDIUM   │ Lock shared       │
│ via shared scope                 │          │ versions, audit   │
├──────────────────────────────────┼──────────┼───────────────────┤
│ XSS in one remote affects        │ HIGH     │ CSP, sanitization │
│ entire host application          │          │ strict mode       │
└──────────────────────────────────┴──────────┴───────────────────┘
```

### Security Best Practices

```
1. HTTPS EVERYWHERE
   - All remoteEntry.js URLs must be HTTPS
   - HSTS headers on all CDNs serving remote assets

2. CONTENT SECURITY POLICY (CSP)
   Content-Security-Policy:
     script-src 'self'
       https://dashboard.myapp.com
       https://reports.myapp.com
       https://settings.myapp.com;
     connect-src 'self' https://api.myapp.com;

3. CORS CONFIGURATION
   Remote CDNs must set:
     Access-Control-Allow-Origin: https://app.mycompany.com
   NOT: Access-Control-Allow-Origin: *

4. SUBRESOURCE INTEGRITY (SRI)
   - Generate SRI hashes for remoteEntry.js in CI
   - Host validates hash before executing remote code
   - Prevents tampered scripts from executing

5. AUTHENTICATION TOKEN HANDLING
   - Use HttpOnly cookies (not accessible to JS)
   - If using localStorage tokens, scope them per remote
   - Never pass tokens through the event bus

6. IFRAME SANDBOXING (for untrusted remotes)
   - Load untrusted remotes in sandboxed iframes
   - Communicate via postMessage with origin validation
   - Provides strongest isolation but worst DX
```

## 16. Tradeoffs & Decision Matrix

### When to Use Module Federation

```
✅ USE Module Federation when:
├── Multiple teams need to deploy independently
├── Application is large enough to justify the complexity
├── Teams have different release cadences
├── You need incremental migration from legacy frameworks
├── Organization has mature CI/CD and DevOps practices
├── Application has clear domain boundaries (dashboard, reports, etc.)
└── Teams are geographically distributed and need autonomy

❌ DO NOT USE Module Federation when:
├── Single team owns the entire application
├── Application is small to medium sized
├── Teams deploy together anyway (shared release train)
├── You don't have the DevOps maturity to manage multiple pipelines
├── Performance is the top priority (MFE adds overhead)
├── You need tight coupling between features (shared state everywhere)
└── Team doesn't understand the complexity tradeoffs
```

## Summary

Module Federation transforms a monolithic frontend into a distributed system of independently
built, deployed, and optionally multi-framework micro-frontends. The host application serves as
the shell and orchestrator, while remotes are loaded at runtime via `remoteEntry.js` manifests.

The key architectural principles are:

1. **Runtime integration** — no build-time coupling between host and remotes
2. **Shared dependency negotiation** — webpack's shared scope prevents duplicate framework bundles
3. **DI hierarchy preservation** — Angular's injector tree works naturally across federated modules
4. **Framework agnosticism** — the DOM is the universal integration layer for multi-framework setups
5. **Contract-based independence** — teams deploy freely as long as they respect the shared contract
6. **Resilience by design** — fallbacks, retries, and graceful degradation for remote failures

The complexity is real, but for organizations with multiple teams, independent release cycles,
and large-scale applications, Module Federation provides the architectural foundation for
sustainable frontend development at scale.
